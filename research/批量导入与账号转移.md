# 批量导入多个邮箱账号 + 账号转移/分享方案

## 1. 一键 / 批量导入多个邮箱账号

### 目标体验
用户有 5~50 个邮箱，想一次搞定，而不是一个个手动填服务器、端口、授权码。

### 推荐方案（分层，兼容不同提供商）

**A. CSV / JSON 批量导入（最实用“一键”方式）**
- UI：设置页 → “批量添加账号” → 上传 CSV 或粘贴表格 / JSON。
- 模板列建议：
  ```
  email, display_name, color, imap_host, imap_port, imap_security (SSL/STARTTLS), smtp_host, smtp_port, smtp_security, auth_type (password | oauth | authcode), secret (授权码或占位), folders (可选: INBOX,Sent), sync_days (默认 90 或 all), priority_weight (0.5~2.0)
  ```
- 示例 CSV（国内 + 国外混用）：
  ```
  work@company.com,工作邮箱,#FF0000,imap.company.com,993,SSL,smtp.company.com,465,SSL,authcode,xxxx-xxxx-xxxx,INBOX,Sent,365,1.5
  personal@gmail.com,个人,#00FF00,,,,,,oauth,, ,INBOX,90,1.0
  notify@163.com,通知订阅,#888888,imap.163.com,993,SSL,smtp.163.com,465,SSL,authcode,yyyy-yyyy-yyyy,INBOX,30,0.3
  ```
- 流程：
  1. 上传/粘贴 → 预览解析结果（自动填充常见提供商服务器，如 163/QQ 自动补全）。
  2. 验证格式 + 冲突检查（重复邮箱）。
  3. 对于 OAuth 账号：弹出逐个授权窗口（或队列引导），用户完成 Google/Microsoft 登录。
  4. 对于授权码账号（163/QQ）：表格中直接填授权码列，或导入后批量弹出输入框。
  5. 干跑（dry run）测试连接（不真正同步）。
  6. 导入 + 开始后台同步（显示每个账号的进度条）。
- 安全：
  - secret 列导入后**立即加密**（使用与 Pebble 类似的设备密钥或用户主密码派生）。
  - 不保存明文 CSV。
  - 支持“只导入设置，不带 secret”（下次使用时提示重新授权）。

**B. 其他批量方式**
- 从现有客户端导出（Thunderbird / Outlook 的 account export，如果有）。
- 从 EmailEngine 实例导入（如果用户之前用过）。
- 粘贴多个 “email:authcode” 行快速添加。
- OAuth 批量：提供一个“连接多个 Google Workspace / Microsoft 365” 入口（如果有 admin consent 权限）。

**C. 国内邮箱特殊优化（必须做好）**
- 预置 163 / QQ / 企业邮 的 imap/smtp 模板。
- 导入时自动提醒：“请先去网页版开启 IMAP 并生成授权码”。
- 一键跳转官方帮助链接。
- 支持“企业微信扫码授权”流程（如果集成企业微信邮箱）。

**技术实现建议**
- 如果后端用 **EmailEngine**：循环调用其 Account registration API（最简单，支持批量）。
- 本地优先（Pebble 风格）：在 Rust 层批量验证 + 加密存储。
- 前端：用 PapaParse 或类似库解析 CSV，TanStack Table 做预览编辑。
- 错误处理：单个账号失败不影响其他；详细错误分类（认证失败 / 服务器拒绝 / TLS 问题） + 修复建议。

**参考**：Mailshake 等工具已有成熟 CSV bulk import 体验；Gmail “Check mail from other accounts” 是简单版单账号拉取。

## 2. 能不能转移邮箱账号给别人？

**直接回答**：**不能直接“转移凭证”**（尤其是 OAuth token 和授权码），但可以安全地**转移配置 + 让接收方重新连接**，或者实现**委托访问（delegation / shared access）**。

### 推荐的转移/分享模式

**模式 1：配置导出 + 接收方重新认证（最安全、最推荐）**
- 导出内容（加密包）：
  - 账号列表（email, display_name, color, imap/smtp 设置, folders, sync policy, priority_weight）。
  - 用户自定义规则、标签、Kanban 卡片、已保存搜索。
  - **凭证可选**：可以导出“加密后的 secret”（接收方需知道导出密码才能解密），或**不导出 secret**（推荐，默认让接收方重新输入授权码 / 重新走 OAuth）。
- 流程：
  1. 原用户：设置 → 导出账号配置 → 选择“包含加密凭证？”（默认否）→ 生成 .zip 或 .pebblebackup 文件（或通过 WebDAV 自动备份如 Pebble）。
  2. 通过安全渠道传给接收方（Signal、加密邮件、U 盘等）。
  3. 接收方导入 → 系统提示“以下账号需要重新授权” → 逐个完成 OAuth 或输入新授权码。
- 优点：凭证从不离开原用户控制范围；支持跨实例/跨设备；Pebbel 已有类似 WebDAV 备份实现。
- 参考：Pebble 的 “可选 local file export/import 和 WebDAV backup for settings, rules, Kanban cards, and separately encrypted account secrets”。

**模式 2：委托访问 / 共享权限（团队/交接场景）**
- 在多用户自托管 Web 版中实现 ACL：
  - 账号所有者可以把某个账号的访问权授予其他用户（只读 / 回复 / 完全控制）。
  - 类似 Gmail “Delegate access” 或 Microsoft Shared Mailbox。
  - 接收方在自己的统一收件箱里看到“共享自 XXX”的账号，操作时使用委托身份。
- 实现：
  - 后端记录 “account_id → shared_with user_id + permission level”。
  - 发送邮件时使用原账号的 SMTP / Send API，但 From 头可配置。
  - OAuth 场景：原所有者授权应用后，应用可代表其操作（需 provider 支持 delegation，如 Microsoft Graph）。
- 适合：企业内部、夫妻/合伙人共用、临时交接。

**模式 3：内容迁移（而非账号本身）**
- 把邮件内容从 A 账号拉到 B 账号（IMAP 拖拽、导出 MBOX/EML/PST 再导入）。
- 聚合工具里提供“一键归档到另一个账号”功能（跨账号移动邮件）。
- 适合“把旧邮箱内容合并到新主邮箱”场景。

**模式 4：完全所有权转移（少见）**
- 接收方用自己的聚合实例重新添加所有账号。
- 原用户在源提供商端撤销旧授权（Google/Microsoft 安全中心、163 第三方管理）。
- 适合彻底 handover。

### 安全与 UX 注意事项（必须强调）
- **永远不要明文分享授权码或 OAuth refresh token**。
- 导出文件必须用强密码加密（AES-256）。
- 导入时显示“此操作将添加以下账号，请确认你是授权接收方”。
- 支持“导出后自动撤销当前实例的本地缓存凭证”（一键交接模式）。
- 日志审计：谁导出了哪些账号的配置。
- 对于国内授权码：导出时可提示“授权码有有效期，接收方需尽快重新生成”。

### 现有工具怎么做
- Pebble：最接近的本地备份 + 加密 secret 分离导出。
- EmailEngine：通过其 API 可以导出/导入账号列表（你的前端控制转移逻辑）。
- 商业团队工具（Missive、Superhuman 团队版）：原生共享收件箱 + 权限管理。
- Outlook / Gmail：委托 + 共享邮箱 + PST 导出导入。

## 建议纳入计划的优先级
- **高优先（MVP 必做）**：CSV 批量导入 + 基础规则引擎（为优先级排序服务）。
- **中优先**：加密配置导出/导入（支持转移） + 清晰的“重新授权”流程。
- **中高优先**（如果做 Web 多用户版）：共享/委托访问 ACL。
- 文档：提供标准 CSV 模板下载 + “如何从 Thunderbird/Outlook 导出账号”指南。

这些功能直接解决“用户很多邮箱时的管理负担”和“账号生命周期管理”问题，是聚合工具从“能用”到“好用”的关键。
