# 自托管、性能、JMAP 与实战补充调研

## 1. EmailEngine 自托管性能关键教训（最重要）
来自官方 Deployment / Performance Tuning / Troubleshooting 文档：

- **Redis 是生命线**：
  - 必须与 EmailEngine 极低延迟（同机 <1ms，同一可用区 <5ms）。
  - 原因：同步文件夹时对**每条消息**执行单独 Redis 操作。大文件夹初始同步可能数千 writes/sec。
  - 高延迟 = 同步极慢甚至超时。
  - 诊断：redis-cli --latency / --latency-history；ping 测试。

- **Redis 配置建议**：
  - maxmemory-policy: noeviction（防止丢索引/令牌导致全量重同步）。
  - 监控 slowlog、keyspace（emailengine scan 命令生成报告）。
  - 持久化注意：初始同步高写入，AOF 可能有问题。
  - 内存：粗略 1-2MB per mailbox（视缓存策略）。

- **无水平扩展**：
  - 不能多实例连同一 Redis（各自独立同步所有账号 → 冲突 + 浪费）。
  - 只能垂直：增加 workers (EENGINE_WORKERS)、CPU、内存。
  - 大规模：手动分片（不同实例 + 不同 Redis DB，由你的应用按账号路由请求）。

- **部署模式对比**（2026 文档）：
  | 方式          | 难度 | 适合          | 备注 |
  |---------------|------|---------------|------|
  | Docker / Compose | 低  | 开发、小规模 | 推荐入门，包含 Redis 示例 |
  | SystemD + Nginx/Caddy | 中 | VPS/裸金属   | Ubuntu 一键脚本好用 |
  | Kubernetes    | 高  | 企业大规模   | 单实例 EmailEngine，K8s 负责 HA/重启 |
  | Render.com 等 | 低  | 快速原型     | 托管，成本随规模上升 |

- **生产 checklist**：
  - Redis 低延迟 + 持久化 + 密码保护。
  - EENGINE_SECRET 强 + 字段加密。
  - Prometheus /metrics + Bull Arena UI 监控队列。
  - 健康检查 /health。
  - 备份策略（尤其是 Redis RDB）。
  - 防火墙 + HTTPS + 限制 localhost 默认访问。

- **故障常见**：
  - Redis 满或 eviction → 重新全同步。
  - 队列堆积（Bull UI 查看）。
  - 原始 IMAP 调试：npm run raw -- --filter.account=xxx。

**结论**：如果你计划用 EmailEngine 做后端，必须在架构图里把“EmailEngine + Redis 同机部署”作为硬性要求，并提供垂直扩展指导。

## 2. JMAP vs 传统 IMAP（现代趋势）
- **JMAP 优势**（JSON over HTTP/WebSocket）：
  - 高效、批处理、实时推送（新邮件瞬间到达）。
  - 状态少、移动/Web 友好。
  - 支持 Mail + Sieve + Quotas + Blob + 更多扩展。
  - Stalwart Mail Server (Rust，开源) 是优秀的全功能实现（JMAP + IMAP + SMTP + CalDAV/CardDAV/WebDAV）。

- **现状 (2026)**：
  - 服务器端：Stalwart 成熟；Fastmail 等商用已用多年。
  - 客户端：仍追赶中。Thunderbird（桌面/Android/iOS）正在积极添加（iOS 已 JMAP-first，桌面 2026 roadmap）。
  - Cypht 已支持 JMAP。
  - 很多自托管者用 Stalwart + 自定义/现有客户端。

- **对聚合项目的意义**：
  - 如果你**同时自建邮件服务器**（全栈邮件方案），JMAP 是未来方向。
  - 如果主要是**聚合外部账号**（Gmail/Outlook/163 等），仍需 IMAP 或 provider API 桥接。JMAP 客户端支持不足时，传统 IMAP 仍是现实选择。
  - 混合方案可行：内部用 JMAP，对外桥接到 IMAP。

- **推荐**：在计划中提及“JMAP 作为未来兼容选项”，但 MVP 以 IMAP + Gmail API + MS Graph 为主。

## 3. 其他自托管 / 性能观察
- **Cypht**：极轻（单页 ~50KB + 并行 AJAX），模块按需加载，内存低。适合资源受限环境。Docker 简单。
- **Pebble (Desktop)**：本地 Tantivy + SQLite 搜索/存储，性能优秀，无服务器端同步瓶颈。网络仅用于实际邮件同步。托盘模式可暂停 worker 省资源。
- **Zero / Inbox Zero**：Postgres + (Redis/Upstash) 典型。Next.js SSR + 后台 worker。AI 调用会增加成本/延迟，需做好队列和限流。
- **通用痛点**（社区）：
  - 统一收件箱更新不及时、重复邮件、文件夹排除问题（Thunderbird 等常见反馈）。
  - Outlook 桌面直到 2025/2026 仍被吐槽缺少好用的统一收件箱 + 全局搜索。
  - 大邮箱初始同步时间长 + 存储占用。
  - 隐私 vs 便利：本地优先（Pebble）天然占优；Web 自托管需认真做凭证加密 + 访问审计。

## 4. 架构建议补充（结合所有调研）
**推荐分层（灵活适配不同计划）**：
1. **连接层**（最重）：EmailEngine（推荐）或自实现 pebble-mail 风格 (IMAP + Gmail API + MS Graph + 国内授权码)。
2. **存储/索引层**：
   - 本地优先：SQLite + Tantivy (Pebble 模式)。
   - Web 自托管：Postgres + Meilisearch/Tantivy/FTS，或 EmailEngine 自身状态 + 你的元数据缓存。
3. **聚合/业务层**：统一消息模型（account_id + normalized headers + thread_id + labels）。
4. **前端/体验层**：React/Next (Zero 风格) 或 Tauri (Pebble 风格)；支持颜色标识账号、按账号过滤、批量操作。
5. **AI 层**（可选增强）：Inbox Zero 风格的 plain-English rules + 语气草稿 + 分类；本地 LLM 或调用 API。
6. **规则/自动化**：客户端规则（快）+ Sieve (如果提供商支持) + AI rules。

**MVP 切分建议**（供参考）：
- Phase 1：添加账号（授权码 + OAuth）→ 基础同步 → 统一列表视图（按时间 + 账号 badge）→ 搜索（先服务器端，后本地索引）。
- Phase 2：发送（选账号）、线程、简单规则/标签、附件预览。
- Phase 3：AI 辅助、Kanban、生产力、跨账号移动、JMAP 实验、国内专项优化、企业微信集成。

**安全/合规**（中国用户特别注意）：
- 凭据：设备密钥加密（桌面）或 at-rest 加密 + 最小权限 OAuth。
- 日志：避免记录敏感内容。
- 自托管 Web：用户数据本地化考虑。
- 备份：重要（尤其是 Redis 或本地 DB）。
